## 1 课题来源、目的及意义

### 1.1 课题来源

随着大模型训练与推理任务规模持续扩大，算力平台从传统 CPU 集群快速向 GPU/NPU 异构集群演进。以 Ascend 平台为代表的新型 AI 计算基础设施在高校和企业中应用日益广泛，但在实际工程中，开发者通常面临以下问题：

1. 性能数据来源分散，系统指标、进程指标、算子级指标分布在不同工具与文件中，难以形成统一视图。  
2. Profiling 结果文件规模大、结构复杂，依赖人工排查，定位瓶颈效率低。  
3. 监控、分析、可视化三者解耦不足，难以形成“采集-诊断-复盘”的闭环。  

本课题来源于实际工程需求：在 Ascend 工具链（CANN/msprof）与 Linux 系统监控基础上，构建一套面向单次任务与多次实验复盘的性能分析可视化系统。项目当前已具备 `imonitor` 与 `analyzer/hprofile` 两条实现基础，具备继续深化为毕业设计系统的条件。

### 1.2 课题目的及意义

本课题目标是设计并实现一套“统一采集、统一建模、深度信息挖掘与可视化展示”的性能分析系统，核心聚焦 Ascend 平台，后续可扩展接入 NVIDIA 工具链。其主要意义如下：

1. 工程价值：降低性能诊断门槛。通过统一数据模型联通系统级、进程级、算子级信息，减少人工翻查多源文件的成本。  
2. 挖掘价值：突破“仅展示原始曲线”的局限，重点挖掘原始数据之间的隐含关系，例如 CPU-NPU 等待关系、GPU/NPU 通信与同步行为、流水线气泡与空转区间。  
3. 方法价值：形成可复用的深度分析流程。将原始 profiling 产物转化为可追溯的结构化关系图谱与报告模板，为后续自动化诊断提供基础。  
4. 学术与实践结合价值：将可观测性工程与性能归因方法引入 AI 推理优化场景，探索异构平台性能分析的体系化实现路径。  
5. 可持续扩展价值：在 Ascend 主线完成后，通过适配层机制对接 NVIDIA Nsight/CUPTI 数据源，实现跨平台分析框架复用。

## 2 国内外研究现况及发展趋势

### 2.1 系统性能监控与可观测性研究现况

国内外在系统监控领域已经形成较成熟方法，典型路线是通过 OS 指标采集、时序存储、聚合分析与图形化展示来支撑运维和性能优化。Prometheus、OpenTelemetry 等体系推动了指标标准化与可观测性工程化，使得 CPU、内存、I/O、网络等指标具备较好的一致性表达[1][2]。在数据库与本地分析侧，SQLite 与列式存储（如 Parquet）被广泛用于轻量级数据管理与离线分析[3][4]。

然而，上述通用监控体系主要面向服务运行状态，缺少对 AI 任务“算子-流-通信-设备”多层语义的直接表达，难以独立完成深度性能归因。

### 2.2 Ascend 及相关工具链研究现况

在 Ascend 生态中，CANN 提供了从运行时到 Profiling 的完整工具链，`msprof` 可采集任务时间线、通信信息、算子统计等关键数据[5][6]。这类工具具备高信息密度，但原始产物目录层次复杂、字段语义分散，通常需要结合日志、导出 CSV 与数据库表进行综合分析。

在 NVIDIA 生态中，Nsight Systems/Nsight Compute/CUPTI 提供了类似的多层性能采集能力[7][8]。从研究与工程实践看，主流思路均强调“高精度采集 + 离线分析 + 可视化解释”，但跨平台统一建模与报告自动化仍是薄弱环节。

### 2.3 发展趋势与问题归纳

综合国内外现状，性能分析系统的发展趋势主要体现在三方面：

1. 从“工具输出”走向“语义表达”：不只展示原始事件，还要提供可解释的阶段划分、等待归因、瓶颈拓扑。  
2. 从“单点监控”走向“跨层联动”：系统资源、运行时行为、算子执行、通信链路需要在同一时间轴上联合分析。  
3. 从“可视化展示”走向“关系挖掘”：重点识别 CPU/GPU（NPU）等待、通信-同步耦合、跨 stream 因果链与流水线气泡。  
4. 从“人工读报告”走向“自动化诊断”：通过结构化数据模型与口径可追溯机制，为规则引擎或智能体分析提供机器可读输入。  

当前仍存在的核心问题是：

1. 数据标准不统一，跨工具链复用成本高。  
2. 现有工具多停留在统计展示，缺少跨数据源关系挖掘与明确证据链。  
3. 对开发者不友好，完整分析流程依赖经验。

因此，围绕 Ascend 工具链实现端到端、可追溯、可扩展的性能分析与深度挖掘系统，具有明确研究与实践意义。

## 3 研究内容与技术方案

### 3.1 课题研究内容

本课题拟完成以下研究与实现工作：

1. 统一采集与接入：面向 Linux 任务运行过程，采集系统与进程资源指标；面向 Ascend profiling 目录，自动发现并读取多 `PROF`/多数据库产物。  
2. 统一数据建模：构建可跨工具链扩展的统一数据模型，覆盖 run 元信息、时间线事件、stream 统计、阶段统计、因果边、质量报告等核心对象。  
3. 深度关系挖掘算法实现：实现 CPU-NPU 等待拆解、通信与同步行为关联、跨 stream 因果边推断、流水线气泡识别、micro-loop 候选挖掘等能力。  
4. 可视化与报告生成：实现任务管理式实时看板与离线增强包（augmentation bundle），用于承载分析结论与证据追溯，而非仅展示原始曲线。  
5. 平台扩展机制设计：通过数据适配层预留 NVIDIA 工具链接入点，验证框架的可迁移性。

### 3.2 课题预期目标

预期目标分为功能目标、质量目标和扩展目标。

1. 功能目标：
- 支持 `imonitor run/inspect/recent/tables/logs` 等核心流程。  
- 支持 Ascend `msprof` 产物自动解析并生成统一报告。  
- 支持 CPU-NPU 等待、通信同步、流水线气泡等高价值关系指标输出。  
- 支持 Web 可视化查询与离线 bundle 交付。  

2. 质量目标：
- 关键指标具备来源追溯能力（lineage）。  
- 数据缺失与匹配率具备质量报告（quality_report）。  
- 在常见开发环境下具备稳定可复现实验流程。  

3. 扩展目标：
- 完成统一 schema 的版本化管理。  
- 定义 NVIDIA 工具链接入所需的最小字段映射规范。

### 3.3 技术路线与方案

#### 3.3.1 关键技术

（1）多源时序数据采集与统一存储技术  
利用 Linux `/proc` 指标采样与可选 NVML 指标采样，结合 SQLite/CSV/Parquet 多种落盘方式，实现“实时可看、离线可算”的数据保存策略。

（2）Ascend profiling 数据解析与语义重构技术  
针对 `msprof_*.db` 与导出摘要文件，完成事件归类、阶段切分、跨流关系重构，并进一步挖掘 CPU-NPU 等待、通信-同步耦合与流水线气泡等隐含行为。

（3）可追溯分析与离线可视化技术  
建立统一 `unified_profile.json`、`lineage.json`、`quality_report.json` 三类核心产物，保证指标口径透明；基于静态 Web 资源承载“挖掘结论 + 数据证据”一体化离线交付。

#### 3.3.2 总体方案

系统总体采用“采集层-建模层-分析层-展示层”的四层架构：

1. 采集层：`imonitor` 启动目标任务并进行周期采样；Ascend 侧通过 `msprof` 生成 run 目录与数据库产物。  
2. 建模层：统一 run 元信息、原始事件与聚合指标，形成标准化数据实体。  
3. 分析层：计算全局占比、stream 画像、阶段统计、因果边与 loop 结构，并输出质量评估。  
4. 展示层：提供实时看板、SQL 查询界面、离线报告与 bundle 文件。

图1 系统总体架构图（后续在论文中绘制）：  
- 上层：Web/UI 与报告导出；  
- 中层：分析引擎（breakdown/causality/loop/quality）；  
- 下层：统一数据模型与多源采集输入（系统采样 + msprof）。

核心指标示例：

\[
R_{wait}=\frac{T_{wait}}{T_{wait}+T_{comm}+T_{exec}+T_{other}}
\]

其中，\(R_{wait}\) 表示等待时间占比，\(T_{wait}\) 等指标由统一时间线聚合得到。

\[
R_{overlap}(i,j)=\frac{\Delta t_{overlap}(i,j)}{\min(\Delta t_i,\Delta t_j)}
\]

其中，\(R_{overlap}(i,j)\) 表示不同 `PROF` 时间窗对齐重叠度，用于判断跨进程时间线联合分析的可信程度。

## 4 可行性与风险分析

### 4.1 制约因素与风险

本课题的主要风险与约束如下：

1. 工具链与环境兼容风险：Ascend 驱动、CANN 版本、容器环境差异可能导致采集结果不一致。  
2. 采样与分析开销风险：高频采样或重型 profiling 开关可能对目标任务产生干扰。  
3. 数据规模风险：长时运行会产生大体量时间线数据，影响解析与可视化响应。  
4. 误判风险：基于启发式的因果推断并非运行时显式依赖，存在解释边界。  
5. 安全与合规风险：日志与运行参数可能包含敏感路径、用户信息，需要最小化暴露。

针对上述风险，拟采取如下措施：

1. 建立版本矩阵与环境快照机制，记录关键版本信息并固化复现实验脚本。  
2. 设置分级采样策略，默认低开销采样；高开销分析按需开启。  
3. 采用“聚合优先、明细分片”策略，限制单次加载的数据规模。  
4. 在报告中明确指标置信边界与数据覆盖率，避免过度结论。  
5. 增加脱敏与访问控制策略，仅开放只读查询与必要字段。

### 4.2 成本估算与可行性

#### 4.2.1 估算模型选择

本课题采用基本 COCOMO 进行成本粗估。原因是：

1. 模型结构清晰，便于在开题阶段快速量化工作量；  
2. 课题规模属于中等复杂度的软件系统，适合半独立型（semidetached）估计；  
3. 估算结果可用于说明“完整工业化系统”与“毕业设计可交付范围”的差异。

#### 4.2.2 参数设置与计算结果

依据项目现有代码统计（仅计入自研实现代码：`imonitor`、`analyzer`、`scripts` 中的 `.py/.sh`，不计入 `msprof-docs-processed/ascend` 爬取文档），规模约为 `10087` 行，折合约 `10.1 KDSI`。采用半独立型基本 COCOMO 公式：

\[
MM=3.0\times(KDSI)^{1.12}
\]

\[
TDEV=2.5\times(MM)^{0.35}
\]

代入 `KDSI=10.1`，得：

1. 工作量 `MM \approx 39.9` 人月；  
2. 开发周期 `TDEV \approx 9.1` 个月；  
3. 平均人员需求约 `4.4` 人。

考虑本课题为单人毕业设计，且已有代码基础可复用，实际采用“聚焦可验证原型”的范围收敛策略：

1. 优先交付 Ascend 主线可运行系统；  
2. 将跨平台扩展控制在接口设计与初步映射，不追求全量实现；  
3. 通过现有工程资产复用降低重复开发成本。

综合评估：在已具备阶段性实现基础的前提下，课题在毕业设计周期内可行。
